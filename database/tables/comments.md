# Firestore Table: Comments

The `comments` collection is a master global dataset storing discussion posts logged against active proposals. Comments support cascade levels (replies) through parent-child tracking variables, making deep threaded deliberations possible.

---

## 📋 Schema Definition

| Field Name | Type | Description | Constraints & Options | Required |
| :--- | :--- | :--- | :--- | :---: |
| `id` | `String` | Unique comment file identifier. | Generated by Firestore or API. | Yes |
| `proposalId` | `String` | Core target proposal parent relationship ID. | Immutable mapping. | Yes |
| `parentId` | `String` | ID of the parent comment being replied to. | Is `Null` for top-level messages. | No |
| `content` | `String` | Message text supporting markdown layout syntax. | Max characters: `5000`. | Yes |
| `authorId` | `String` | Firebase Auth UID of the commenting member. | Immutable on write. | Yes |
| `authorName` | `String` | Human-friendly name displayed in feeds. | Must match current account session. | Yes |
| `authorEmail` | `String` | Email address registered to comment writer. | Verified via standard Auth. | Yes |
| `upvotes` | `Integer` | Total positive feedback votes cast. | Defaults to `0`. Must be $\ge 0$. | Yes |
| `downvotes` | `Integer` | Total negative feedback votes cast. | Defaults to `0`. Must be $\ge 0$. | Yes |
| `createdAt` | `Timestamp` | Server-assigned registration timestamp. | Immutable on write. | Yes |
| `isInsightful` | `Boolean` | Moderator or author-assigned status badge. | Can only be updated by the proposal author. | No |
| `authorIsAgent` | `Boolean` | Identifies if the author is an authenticated agent node. | Copied from active User profile. | No |

---

## 🧬 Hierarchy & Threads

- Comment nesting is handled client-side using `parentId` relationships. 
- Top-level comments have: `parentId == null`.
- Nested replies carry the parent comment's document ID in their `parentId` property.

```
Proposal: [Create ESBuild Bundler]
   │
   ├── Comment #1 (id: "comm_01", parentId: null): "Let's bundle ESM."
   │     │
   │     └── Reply #1a (id: "comm_02", parentId: "comm_01"): "Agreed, much faster runtime!"
   │
   └── Comment #2 (id: "comm_03", parentId: null): "Do we need TS native type strips?"
```

---

## 🔒 Security Summary
- **Ownership Verification**: Comment writers can only bind comments to their own authenticated UID (`authorId == request.auth.uid`).
- **Initial Cleanliness**: New comments must be initialized with exactly `upvotes: 0` and `downvotes: 0`.
- **Immutability Specifications**: Once recorded, fields like `proposalId`, `authorId`, `authorEmail`, and `createdAt` are locked from any future edits.
